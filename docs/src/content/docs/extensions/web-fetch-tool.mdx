title: web-fetch-tool
description: "Fetch web content with context-budget-aware output caps"
---

`web_fetch` retrieves a URL over HTTP and returns text output.

The tool now uses **context-budget-aware caps** so a large fetch does not blow up a single turn.

## Parameters

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `url` | string | — | URL to fetch |
| `maxBytes` | number | adaptive | Hard upper bound for tool output; may be reduced by safety budget |
| `format` | string | `"text"` | Output hint: `"text"`, `"markdown"`, or `"html"` |

## Adaptive budget behavior

At execution time, `web_fetch` resolves `effectiveMaxBytes` from four inputs:

1. **Planner envelope** (if available)
2. **Policy bounds** (min/max per tool)
3. **User `maxBytes`** (hard upper bound)
4. **Strict fallback** when no envelope is available

Rules:

- User `maxBytes` is always an upper bound.
- Planner envelopes can reduce output further for safety.
- Unknown usage states use conservative fallback caps.

## Default policy values

| Setting | Default |
|---------|---------|
| minimum per-tool bytes | `4096` |
| maximum per-tool bytes | `524288` |
| unknown-usage fallback cap | `32768` |

## Tool result details

`web_fetch` includes audit fields in `details`:

- `effectiveMaxBytes`
- `budgetLimited`
- `budgetReason`
- `batchSize`

These make it clear why truncation happened and whether a planner envelope was applied.

## When to use

- Reading documentation or articles
- Checking API responses
- Inspecting page text from a known URL
